# 需求

## 1. 功能需求

### 1.1 TCX 輸入

工具必須支援：

- 單一 `.tcx` 檔案作為輸入。
- 包含多個 `.tcx` 檔案的資料夾作為輸入。
- 在資料夾模式中忽略非 TCX 檔案，或對其提出警告。

工具不得修改原始 TCX 檔案。

### 1.2 活動範圍

MVP 必須支援 Running 活動。

如果輸入 TCX 包含其他活動類型：

- 工具應在可行時偵測出來。
- 工具不應默默把它當成 Running 處理。
- MVP 中可以略過該檔案，並給出清楚的警告。

### 1.3 解析資料

可用時，工具應解析以下活動層級欄位：

- 運動類型
- 活動 ID 或開始時間
- 開始時間
- 總時間，單位為秒
- 距離，單位為公尺
- 卡路里
- 平均心率
- 最大心率
- 最大速度

可用時，工具應解析以下單圈層級欄位：

- 單圈開始時間
- 總時間，單位為秒
- 距離，單位為公尺
- 卡路里
- 平均心率
- 最大心率
- 最大速度
- 強度
- 觸發方式

可用時，工具應解析以下 trackpoint 層級欄位：

- 時間戳記
- 緯度
- 經度
- 海拔，單位為公尺
- 距離，單位為公尺
- 心率
- 速度
- 跑步步頻
- 功率

### 1.4 輸出格式

每個轉換後的活動，工具必須產生：

- `activity.json`
- `trackpoints.csv`
- `ai_summary.json`
- `ai_summary.md`

在批次模式中，每個 TCX 檔案都應產生自己的輸出檔案組。

### 1.5 AI-ready 摘要

AI-ready 摘要必須包含：

- 基本活動中繼資料
- 距離
- 持續時間
- 平均配速
- 平均心率
- 最大心率
- 海拔摘要
- 單圈摘要
- 前半段與後半段的配速趨勢
- 前半段與後半段的心率趨勢
- 使用的 GPS 隱私政策
- 缺漏資料的說明

AI-ready 摘要必須避免做出醫療或專業教練聲明。

趨勢計算規則：

- 可行時，使用累積距離切分前半段與後半段。
- 若距離、時間或 trackpoint 資料不足，趨勢應標示為
  `insufficient_data`。

### 1.6 缺漏資料

如果標準欄位缺漏：

- JSON 輸出應使用 `null`。
- CSV 輸出應保留空白儲存格。
- AI 摘要只在缺漏資料影響解讀時提及。

如果出現未知或不支援的擴充欄位：

- 工具可以略過該欄位。
- 可行時，工具應記錄警告。
- 轉換應繼續進行。

## 2. 非功能需求

### 2.1 可維護性

本專案必須能由一位開發者維護。

實作應偏向：

- 小模組
- 清楚的函式邊界
- 明確的資料契約
- 最少依賴
- 不使用不必要的框架

### 2.2 Python 標準

未來 Python 程式碼必須遵循：

- PEP8 風格
- 公開函式與資料結構要有 type hints
- 模組、類別與公開函式要有 docstrings
- 命名清楚；避免單字母變數名稱

### 2.3 本機優先的資料處理

工具必須能在本機運作。

MVP 不得：

- 將資料上傳到雲端服務。
- 向 Garmin Connect 進行驗證。
- 將資料儲存在資料庫。
- 直接將資料送到 AI APIs。

### 2.4 隱私

工具必須將 GPS 與健康資料視為敏感資料。

必要的隱私行為：

- 原始 TCX 檔案是唯讀輸入。
- GPS policy 必須明確，並記錄在 AI-ready 輸出中。
- 心率、配速與距離會保留，除非未來設定另有規定。
- 文件必須說明與 AI 應用程式分享 GPS 與健康資料的風險。

### 2.5 錯誤處理

錯誤訊息應可讀且可採取行動。

工具應區分：

- 找不到檔案
- 不支援的檔案類型
- 無效 XML
- 有效 TCX，但活動類型不支援
- 缺漏選填資料
- 輸出寫入失敗

警告物件必須包含：

- `code`
- `severity`
- `field`
- `message`
- `source_file`

CLI exit code 慣例：

- `0`：所有轉換都成功。
- `1`：批次已完成，但至少一個檔案失敗或被略過。
- `2`：轉換開始前發生使用者、設定或輸入錯誤。

## 3. MVP 邊界

MVP 不得實作：

- Web UI
- SQLite 或其他資料庫
- GarminDB 整合
- python-garminconnect 整合
- Garmin Connect 登入
- NotebookLM 或 ChatGPT API 上傳
- 完整訓練建議引擎

這些只能出現在路線圖文件中。
